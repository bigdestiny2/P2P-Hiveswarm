# HiveRelay Security + Functional Audit
Date: 2026-04-15
Audited commit baseline: `0870c6e` plus local hardening changes in working tree

## Executive Summary

HiveRelay is strongest today as an availability and reachability layer for P2P apps:
- relay seeding for uptime,
- NAT fallback via circuit relay,
- HTTP gateway + catalog for discovery.

Security posture improved materially in this pass (auth gates, CORS default deny, route-level access classes, private-mode admission control), but several product claims remain overstated versus runtime guarantees, especially distributed registry behavior and privacy enforcement semantics.

## Findings

## [P1] Distributed seeding registry is still not actually distributed
- File: `core/registry/index.js`
- Evidence:
  - peer log tracking is declared but not used (`peerLogs` map): lines 29, 203-208
  - startup indexes only local log: lines 40-45
  - connection path only replicates local log, no remote log discovery/indexing: lines 58-61
- Impact:
  - registry behaves like local append-only state per node, not network-merged truth.
  - claims about global request/acceptance consistency are overstated.

## [P1] Privacy tier enforcement is partial, not end-to-end fail-safe
- Files:
  - `core/relay-node/index.js`
  - `core/services/builtin/storage-service.js`
- Evidence:
  - seeding path replicates and serves drives without a tier gate in `seedApp`: lines 669-756
  - PolicyGuard checks are currently service-operation scoped in storage service only: `storage-service.js` lines 151-158
- Impact:
  - PolicyGuard exists and works for storage RPC flows, but not as a full relay-wide seeding/storage admission system.
  - “fail-safe privacy enforcement” remains only partially true.

## [P2] Route authorization model is heuristic, not declarative per route
- File: `core/router/index.js`
- Evidence:
  - access class inference is method-name driven (`_inferAccessPolicy`): lines 160-183
- Impact:
  - new or renamed route methods can get unintended access class if naming pattern changes.
  - should move to explicit per-service/per-route policy maps for production assurance.

## [P2] Payments/economics remain opt-in and backend-dependent
- File: `core/relay-node/index.js`
- Evidence:
  - settlement loop only starts when both `payment.enabled` and external `paymentManager` are provided: lines 404-412
- Impact:
  - monetization/trust/economics claims are not default runtime behavior.
  - operator economics should remain “experimental/optional” in external positioning.

## Functional Reality Matrix

### Strong / production-usable now
- Core relay seeding and persistence
- Circuit relay fallback for NAT failures
- Catalog + gateway baseline
- Basic service dispatch framework
- Management and health endpoints
- Private-mode connection allowlist enforcement (with pairing primitives)

### Partial / still maturing
- Privacy model guarantees beyond storage-service checks
- HomeHive/private mode product completeness (discovery UX, tunnel orchestration, operational polish)
- Service authorization as policy-driven RBAC
- Registry federation/replication correctness
- SLA/payments trust layer for production buyers

## Value Proposition (Current Truth)

Best one-line wedge:
**HiveRelay keeps P2P apps available and reachable when normal peer availability fails.**

What this is worth:
- immediate reliability gain for Pear/Hyperswarm developers,
- less infra burden (no bespoke backend needed for uptime/discovery),
- practical bridge for browser/NAT-constrained users.

What should not be over-promised yet:
- “fully distributed registry consensus”
- “fail-safe privacy tier enforcement across all relay flows”
- “production-grade economics/SLA guarantees by default”

## Changes Implemented In This Audit Pass

- Added mode-profile configuration system with validation in `RelayNode`:
  - supports `public`, `private`, `hybrid`, `homehive`, `standard`, `seed-only`, `relay-only`, `stealth`, `gateway`
  - nested `discovery` / `access` / `pairing` defaults merged safely with overrides.
- Added route-level access classes in router:
  - `public`, `authenticated-user`, `relay-admin`, `local-only`
  - dispatch now denies unauthorized access before handler execution.
- Updated service protocol context to pass authenticated user role for remote P2P requests.
- Hardened API:
  - auth on `/api/v1/dispatch`
  - auth on registry mutation and registry introspection endpoints
  - local-only route allowance only for localhost
  - deny-by-default CORS preflight behavior
  - explicit 400 for invalid JSON and 413 for oversized bodies.
- Extended tests:
  - route access policy tests
  - updated dispatch auth behavior tests
  - private-mode config and admission tests now pass.
