Quick Start

Get a HiveRelay node running in under a minute.

Installation

bash
npm install p2p-hiverelay

Run a Relay Node

javascript
import { RelayNode } from 'p2p-hiverelay'

const node = new RelayNode({
  storage: './relay-data',
  apiPort: 9100,
  regions: ['NA'],
  enableSeeding: true,
  enableRelay: true
})

await node.start()
console.log('Relay running! Public key:', node.swarm.keyPair.publicKey.toString('hex'))

Or via CLI

bash
npx hiverelay start --region NA --port 9100
Tip
Once running, open http://localhost:9100/dashboard to see the node operator dashboard, or /network for the full network overview.

Systemd Setup (Production)

For production Linux deployments, use a systemd unit file:

ini
[Unit]
Description=HiveRelay P2P Relay Node
After=network-online.target

[Service]
Type=simple
ExecStart=/usr/bin/node /opt/hiverelay/cli/index.js start --region NA --port 9100
WorkingDirectory=/opt/hiverelay
Restart=always
RestartSec=5
Environment=NODE_ENV=production
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
bash
# Install and enable the service
sudo cp hiverelay.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now hiverelay

Multi-Instance Setup

Run multiple relay instances on the same machine with separate ports and storage directories:

bash
# Instance 1 - North America
npx hiverelay start --region NA --port 9100 --storage ./relay-na

# Instance 2 - Europe
npx hiverelay start --region EU --port 9101 --storage ./relay-eu

# Instance 3 - Asia Pacific
npx hiverelay start --region AP --port 9102 --storage ./relay-ap

Auto-Accept vs Approval Mode

By default, relays run in auto-accept mode — all incoming seed requests are accepted automatically. This is best for open community relays.

For curated relays, switch to approval mode:

  • Set registryAutoAccept: false in config
  • Or toggle via the dashboard UI
  • Or call POST /registry/auto-accept with { "enabled": false }

In approval mode, seed requests appear under /api/registry/pending and must be manually approved or rejected by the operator.

Pear Developer SDK Guide

Installation

bash
npm install p2p-hiverelay

Basic Usage

Create a client, connect to the relay network, and publish content that will be seeded automatically:

javascript
import { HiveRelayClient } from 'p2p-hiverelay/client'

const client = new HiveRelayClient('./app-storage', {
  maxRelays: 5,
  seedReplicas: 2,
  autoSeed: true
})

client.on('relay-connected', (evt) => console.log('Connected to relay:', evt.pubkey.slice(0, 12)))
client.on('seeded', (evt) => console.log('App seeded by', evt.acceptances, 'relays'))

await client.start()

Publish Files to a Hyperdrive

javascript
const drive = await client.publish([
  { path: '/index.html', content: '<h1>Hello World</h1>' },
  { path: '/app.js', content: 'console.log("running")' }
])

console.log('Drive key:', drive.key.toString('hex'))

Read Back

javascript
const data = await client.get(driveKey, '/index.html')
const files = await client.list(driveKey, '/')

Seed Requests

Request relays to persist your app across the network:

javascript
await client.seed(driveKey, { replicas: 3 })
Note
The client will find available relays via DHT discovery and request seeding from enough relays to satisfy the replica count. If a relay goes offline, the client will automatically find replacements.

Architecture

How Discovery Works

HiveRelay uses a fully decentralized discovery model. Clients and relays both join a well-known Hyperswarm DHT topic. No central registry or bootstrap server is needed.

  • Relays announce themselves as servers on the discovery topic
  • Clients join as clients and discover relays automatically
  • Once connected, a Protomux handshake establishes encrypted channels

Discovery Flow

Client ➔ DHT Lookup ➔ Find Relay Peers ➔ Protomux Handshake ➔ Seed Request / Accept

Dual-Path Seeding

HiveRelay uses two complementary paths to ensure seed requests always reach relays:

Path 1: Protomux (Instant, Direct P2P)

Real-time seed requests sent directly over Protomux protocol channels. This is the fast path — if the relay is online, it gets the request immediately.

Path 2: Seeding Registry (Persistent)

A Hypercore-based append-only log that acts as a catch-up mechanism. If a client publishes while a relay is offline, the relay picks it up from the registry when it comes back online. This ensures no seed request is ever lost.

Why dual-path?
P2P networks are inherently unreliable — peers come and go. The dual-path design ensures that seed requests survive relay downtime. Protomux handles the happy path; the registry handles everything else.

Self-Healing

Every relay node runs a continuous health monitor that checks:

  • Memory usage — restarts subsystems if heap exceeds thresholds
  • Connection state — detects and recovers from swarm disconnections
  • Error rates — tracks error frequency and triggers self-heal if rates spike
  • Swarm health — monitors DHT connectivity and re-announces if needed

When issues are detected, the health monitor automatically restarts the affected subsystem without taking the entire node offline.

Network & Discovery

HiveRelay's network is fully decentralized. Here's how it works:

DHT Discovery Topic

All relays and clients share a well-known discovery topic: hiverelay-discovery-v1. This topic is derived deterministically, so every participant finds each other without any configuration.

Seeding Registry Topic

The persistent seeding registry uses the topic hiverelay-seeding-registry-v1 for its Hypercore-based append-only log.

Key Properties

  • No manual peer configuration — relays announce, clients discover automatically
  • No central server — the DHT is fully distributed across all participants
  • NAT traversal — Hyperswarm handles UDP hole-punching for peers behind NATs
  • Real-time network view — the network dashboard shows all discovered relays live
Firewall Note
For best connectivity, ensure your relay can accept inbound UDP and TCP connections. While Hyperswarm can hole-punch through most NATs, a publicly reachable relay provides the best experience for all peers.

API Reference

All endpoints are served by the relay node's built-in HTTP API on the configured apiPort.

GET /api/overview

Returns node stats including uptime, connections, storage, and memory.

json
// Response
{
  "uptime": 86420,
  "connections": 23,
  "storageUsedBytes": 1073741824,
  "memoryMB": 128,
  "publicKey": "a1b2c3d4...",
  "regions": ["NA"],
  "version": "1.0.0"
}

GET /api/network

Returns all discovered relays across the DHT.

json
// Response
{
  "relays": [
    {
      "publicKey": "a1b2c3d4...",
      "regions": ["NA"],
      "connections": 15,
      "uptime": 172800,
      "seededApps": 42,
      "status": "online"
    }
  ],
  "totalRelays": 8,
  "onlineRelays": 6
}

POST /registry/publish

Publish a seed request to the registry.

json
// Request body
{
  "driveKey": "f9e8d7c6...",
  "replicas": 3
}

// Response
{ "ok": true, "id": "req_abc123" }

POST /registry/approve

Approve a pending seed request.

json
// Request body
{ "id": "req_abc123" }

// Response
{ "ok": true }

POST /registry/reject

Reject a pending seed request.

json
// Request body
{ "id": "req_abc123" }

// Response
{ "ok": true }

POST /registry/auto-accept

Toggle auto-accept mode on or off.

json
// Request body
{ "enabled": false }

// Response
{ "ok": true, "autoAccept": false }
